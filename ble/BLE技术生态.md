## BLE 协议栈

### 完整层模型

```
┌─────────────────────────────────────────────┐
│           应用层                             │
│  (应用：健康、健身、配网)                    │
└────────────────┬────────────────────────────┘
                 │
┌────────────────▼────────────────────────────┐
│   配置文件层 (GATT)                          │
│  ├─ GAP：通用访问配置文件                   │
│  ├─ GATT：通用属性配置文件                  │
│  ├─ SDP：服务发现协议                       │
│  └─ 自定义配置文件                          │
└────────────────┬────────────────────────────┘
                 │
┌────────────────▼────────────────────────────┐
│   中间件协议                                 │
│  ├─ ATT：属性协议                           │
│  │  └─ 请求/响应、客户端/服务器            │
│  ├─ SMP：安全管理器协议                     │
│  │  └─ 配对、绑定、加密                    │
│  └─ L2CAP：逻辑链路控制和适配              │
│     └─ 分段、重组、QoS                     │
└────────────────┬────────────────────────────┘
                 │
┌────────────────▼────────────────────────────┐
│   链路层（HCI - 主机控制器接口）            │
│  ├─ 广播和扫描                             │
│  ├─ 连接管理                               │
│  ├─ 数据通道（37 个数据通道）              │
│  ├─ 链路加密 (AES-CCM)                     │
│  └─ 频率跳变扩频 (FHSS)                    │
└────────────────┬────────────────────────────┘
                 │
┌────────────────▼────────────────────────────┐
│   物理层（射频）                             │
│  ├─ 2.4 GHz ISM 频段（无许可）             │
│  ├─ 40 个信道（3 个广告 + 37 个数据）      │
│  ├─ 1 Mbps (LE 1M PHY)                    │
│  ├─ 2 Mbps (LE 2M PHY, BLE 5.0+)         │
│  ├─ 125 kbps (LE Coded, BLE 5.0+)        │
│  └─ +4 dBm 至 +20 dBm 发射功率           │
└─────────────────────────────────────────────┘

主机（软件 - Android）
───────── HCI 接口 ─────────
控制器（固件 - 蓝牙芯片）
```

### ATT（属性协议）

```
属性 = GATT 中最小信息单位

结构：
┌─────────────────┐
│ 句柄 (16 位)    │  唯一标识符 (0x0001-0xFFFF)
├─────────────────┤
│ UUID (128 位)   │  服务/特征/描述符类型
├─────────────────┤
│ 权限            │  READ、WRITE、NOTIFY 等
├─────────────────┤
│ 值 (0-512B)     │  实际数据
└─────────────────┘

操作：
- Find：通过 UUID 查找句柄
- Read：通过句柄读属性值
- Write：通过句柄写属性值
- Notify：服务器→客户端（单向）
- Indicate：服务器→客户端（需 ACK）
- Signed Write：无加密但有签名的写
```

### L2CAP（逻辑链路控制和适配）

```
函数：
1. 分段和重组
   ├─ 将大数据分解为小 PDU
   └─ 重新组装接收的 PDU

2. 多路复用
   ├─ 单个物理链路上的多个逻辑通道
   └─ 属性、信令等的独立通道

3. 流控制
   ├─ 基于信用的流控制
   └─ 防止缓冲区溢出

4. 服务质量 (QoS)
   ├─ 延迟要求
   └─ 吞吐量要求

示例：通过 ATT 写入 100 字节
L2CAP 层：
- 分段为 ~20 字节块
- 为每个块添加 L2CAP 头
- 通过链路层发送
- 链路层 ACK 每个数据包
- 接收器在 L2CAP 中重组
- 将完整消息传给 ATT

结果：对应用程序透明（库处理）
```

### SMP（安全管理器协议）

```
职责：
1. 配对
   ├─ 交换能力
   ├─ 选择配对方法
   └─ 生成或交换密钥

2. 绑定
   ├─ 持久存储配对密钥
   └─ 识别已绑定设备

3. 加密
   ├─ 启用/禁用链路加密
   ├─ AES-CCM 密码套件
   └─ 防止窃听

4. 认证
   ├─ 验证设备身份
   └─ 防止 MITM 攻击

密钥类型：
- 临时密钥 (TK)：配对期间生成
- 长期密钥 (LTK)：绑定后存储
- 完整性检查密钥 (IRK)：解析私有地址
- 连接签名解析密钥 (CSRK)：签名写
```

## BLE 5.x 新特性

### BLE 5.0（2016）- 主要功能扩展

#### 1M/2M PHY（物理层模式）

```
LE 1M（原始）：
- 1 Mbps 数据速率
- 良好范围：100+ 米视线
- 大多数 BLE 设备标准

LE 2M（BLE 5.0+）：
- 2 Mbps 数据速率（2 倍快）
- 更好的吞吐量
- 范围较短（某些设备）
- 并非所有芯片支持

LE Coded（BLE 5.0+）：
- 125 kbps 或 500 kbps（长距离）
- 4-6 倍更长范围：400+ 米
- 数据速率低（较慢）
- 功耗更高
- 用例：建筑规模覆盖

Android 选择：
- 大多数情况自动协商
- setPreferredPhy() 请求特定 PHY
- onPhyUpdate() 回调协商时调用
```

#### 扩展广告

```
经典广告（31 字节限制）：
广告 PDU 结构：
┌──────────────────────────────┐
│ 标志 (1-3 字节)              │
│ 完整 UUID 列表               │
│ 设备名称                      │
│ 制造商数据                    │
│ 总最多 31 字节                │
└──────────────────────────────┘

问题：无法容纳大量数据（如完整传感器读数）

扩展广告（BLE 5.0+，每数据包 251 字节）：
LE 广告数据 (AD) 可包括：
- 状态信息
- 传感器读数
- Beacon 数据
- 更完整的元数据

缺点：需要 BLE 5.0+ 兼容接收器
解决方案：发送多个广告数据包

配网中的用例：
- 在扩展广告中包含设备信息
- 客户端连接前查看设备状态
- 示例："WiFi 配网就绪，电池 85%"
```

#### 周期性广告

```
用例：无连接常规广播

传统：
┌──────┐
│Adv1  │ 等待 100ms
├──────┤
│Adv2  │ 等待 100ms
├──────┤
│Adv3  │ ...
└──────┘

周期性（BLE 5.0+）：
┌──────┐
│Sync1 │ 等待 1 秒（可配置）
├──────┤
│Sync2 │ 等待 1 秒
├──────┤
│Sync3 │ ...
└──────┘

优势：
- 功耗更低：可预测的同步间隔
- 可靠性更好：客户端期望数据在间隔处
- 无连接开销

用例：无头设备广播配网
```

### BLE 5.1（2019）- 方向和定位

```
方向查找（角度到达 - AoA）：
- 使用天线阵列确定信号方向
- 可定位 BLE 设备，精度达到米级
- 用例：室内导航、资产跟踪

角度出发 (AoD)：
- 设备传输参考信号
- 接收器计算角度
- 用于基于方向的认证
```

### BLE 5.2（2021）- LE 音频和等时

```
LE 音频（新）：
- 编码：LC3（低复杂度通信编码）
- 比旧编码码率更低、质量更好
- 支持助听器、耳机
- Qualcomm Snapdragon X、Apple H1 芯片等

等时通道：
- 时间敏感型数据传输
- 保证带宽
- 用例：音频流、实时视频
- 低延迟比 GATT

对配网的影响：
- 最小（配网不时间关键）
- 可启用配网期间的音频反馈
```

### BLE 5.3（2021）- 广播器扩展

```
周期广告同步传输 (PAST)：
- 中心可将周期广告传输给其他设备
- 同步数据分发
- 用例：批量配网（一个同步对多个设备）

增强隐私：
- 更好的随机地址处理
- 设备身份抗跟踪
```

## BLE Mesh

### 什么是 BLE Mesh？

```
传统 BLE：
设备 A ←BLE→ 中心集线器 ←BLE→ 设备 B
             （瓶颈）

BLE Mesh（SIG 标准）：
设备 A
    ├─ Mesh ─→ 设备 B
    └─ Mesh ─→ 设备 C（中继到其他设备）
                ├─ Mesh ─→ 设备 D
                └─ Mesh ─→ 设备 E
```

### Mesh 架构

```
角色：
1. 配网器 (Provisioner)
   - 向 Mesh 添加新设备
   - 分配地址和密钥
   - 配置设备

2. 节点 (Node/Device)
   - 加入 Mesh 网络
   - 参与消息路由
   - 可为中继、中继禁用或中继启用

3. 中继节点 (Relay Node)
   - 转发消息到其他设备
   - 扩展范围：每跳 ~10 米
   - 典型网络：30+ 米覆盖

4. 低功耗节点 (LPN)
   - 电池供电
   - 不中继消息
   - 定期唤醒检查消息

5. 友元节点 (Friend Node)
   - 为 LPN 存储消息
   - LPN 消息到达时唤醒
```

### Mesh 配网

```
新设备：未配网
     │
     ├─ 广播："我是未配网设备"
     │
配网器（手机）：
     ├─ 接收广播
     ├─ 发起配网
     │
设备 + 配网器：
     ├─ 带外认证（PIN、按钮）
     ├─ 交换密钥
     ├─ 配网器分配地址 + 网络密钥
     │
设备：已配网
     ├─ 现在是 Mesh 的一部分
     ├─ 可接收中继消息
     └─ 可为其他设备中继

流程：
1. 扫描未配网设备
2. 选择设备
3. 认证（OOB 方法）
4. 分配地址 (0x0001-0xFFFF)
5. 分发网络密钥
6. 分发应用密钥（订阅）
7. 成功 → 设备已配网
```

### 一键配网（用例）

```
传统 WiFi 配网：
- 一次配网一个设备
- 用户：1 个设备，30 秒 = 30 秒
- 用户：100 个设备，每个 30 秒 = 50 分钟 ❌

Mesh 配网：
- 配网根设备 → 成为配网器
- 配置中继 → 网络扩展
- 配网多个设备 → 通过 Mesh 中继
- 用户：1000 m² 跨 100 个设备 = 15 分钟 ✓

Mesh WiFi 配网（混合）：
1. 配网设备 A（通过 BLE 或 Mesh）
2. 设备 A 连接到 WiFi
3. 设备 A 成为协调器
4. 其他设备通过 Mesh 配网
5. 协调器将 WiFi 凭证中继给所有
6. 所有设备连接到同一 WiFi 网络
```

### 限制

- 协议更复杂（简单配网过度设计）
- 并非所有芯片支持（需要 Mesh 兼容硬件）
- 电池消耗更长（持续路由）
- 对于一次配网的静止设备过度设计

## 安全机制

### BLE 配对方法

```
┌──────────────────────────────┐
│ BLE 配对方法                  │
├──────────────────────────────┤
│ 1. Just Works                │
│    - 无用户交互              │
│    - 6 位 PIN 交换           │
│    - 易受 MITM 攻击          │
│    ✓ 快速、简单              │
│    ✗ 安全性较低              │
│                              │
│ 2. Passkey Entry             │
│    - 用户在一个设备上输入 PIN│
│    - 其他设备显示相同 PIN    │
│    - 比 Just Works 更安全    │
│                              │
│ 3. 数字比较                   │
│    - 两个都显示 6 位数字     │
│    - 用户确认匹配             │
│    - 最快的安全配对           │
│                              │
│ 4. 带外 (OOB)                │
│    - 预共享密钥              │
│    - 通过 NFC、QR 代码等交付 │
│    - 最高安全性              │
│                              │
│ 5. 遗留配对                   │
│    - BLE 4.0-4.1             │
│    - 较弱（被 LE 安全连接替代）
└──────────────────────────────┘

对于配网用例：
- 无人设备：Just Works
- 有 UI 的消费品：数字比较
- 高安全性：OOB（QR 代码扫描）
```

### 地址随机化

```
私有 BLE 地址：
├─ 静态随机地址
│  └─ 生成一次，持久存在
│     用途：无 EIRP 的设备（闪存存储）
│
├─ 可解析私有地址 (RPA)
│  └─ 每 15 分钟改变
│     用途：关注隐私的设备
│     可通过 IRK 解析（身份解析密钥）
│
└─ 不可解析私有地址
   └─ 每 15 分钟改变
      无法链接到身份
      最大隐私但难以识别

配网影响：
- 可能看到同一设备的多个地址
- 使用 UUID 或其他标识符，不用 MAC 地址
- RPA 对隐私关注的 IoT 有用
```

### 链路加密 (AES-CCM)

```
AES-CCM（CCM 模式的计数器）：
├─ 128 位加密（AES 标准）
├─ CCM 模式提供认证 + 加密
└─ 防止窃听和篡改

加密过程：
1. 配对期间交换长期密钥 (LTK)
2. 绑定后用 LTK 加密链路
3. 所有数据加密传输
4. 接收器解密并验证 MAC

配网影响：
- 启用配对：凭证加密
- 无配对：凭证明文
- 推荐：启用配对以确保安全
```

### 会话密钥推导 (ECDH)

```
BLE 4.2+（LE 安全连接）：
- 使用 ECDH（椭圆曲线迪菲赫尔曼）
- 替换旧的较弱密钥推导
- 从共享密钥生成唯一会话密钥

过程：
1. 配对开始
2. 两者生成 ECDH 公钥
3. 交换公钥
4. 计算共享密钥（私密）
5. 从密钥推导会话密钥
6. 不需要预共享密钥

优势：
✓ 抗 MITM 攻击
✓ 前向保密（旧密钥不影响新会话）
✓ 现代密码学
```

## Android BLE 已知问题

### 问题 #1：GATT 错误 133（连接丢失）

**症状**：`onConnectionStateChange(status=133, newState=0)`

```
原因：设备异常断开连接或
       系统蓝牙崩溃

受影响：Nexus 5、Galaxy S4、Samsung Galaxy Tab（随机）

解决方案：
1. 立即重连带延迟
   ├─ 等待 500-1000ms
   ├─ 尝试重连，autoConnect(false)
   └─ 重试 3-5 次

2. 清除 BluetoothGatt 资源
   ├─ 立即 gatt.close()
   ├─ 创建新 BluetoothGatt
   └─ 尝试重连

3. 重启蓝牙
   ├─ 最后手段
   ├─ 切换蓝牙关闭/打开
   └─ 重连

实现（Nordic 库）：
connect(device)
    .retry(3, 200)  // 自动处理
    .timeout(15_000)
    .enqueue()
```

**为什么 Nordic BLE 库重要**：
- 自动处理重连逻辑
- 管理 GATT 资源生命周期
- 重试指数退避

### 问题 #2：onCharacteristicChanged 多线程 Bug

**症状**：特征改变时 `ConcurrentModificationException`

```
根本原因：特征值存储在数组中
            回调线程 + UI 线程修改

场景：
线程 1（BLE）：[Char1、Char2、Char3]
                ├─ onCharacteristicChanged() 迭代
                └─ 修改 Char2.value

线程 2（主线程）：
                ├─ 读特征列表
                └─ ❌ 并发修改！

解决方案：Android 7.0+ 提供线程安全处理
          Nordic 库抽象这一点

Android 6 及以下：
- 在本地变量中存储值
- 发布到主线程后访问
- 避免直接特征数组访问
```

### 问题 #3：扫描频率限制 (Android 7.0+)

**症状**：在快速连续进行多次扫描后扫描停止工作

```
策略：最多 5 次 BLE 扫描每 30 秒
      （在后台/瞌睡模式中）

受影响：后台扫描耗电

解决方案：
1. 使用前台服务（可见通知）
   - 豁免扫描限制
   - 总是工作

2. 使用 WorkManager 定期扫描
   - 批量扫描
   - 尊重瞌睡模式

3. 手动限制
   - 应用：扫描会话间等待 30 秒
   - 跟踪上次扫描时间
   - 不在 < 30 秒时启动新扫描

Android 12+ 位置模式：
- 高精度：允许扫描
- 省电：限制
- 位置关闭：后台无 BLE 扫描

实现：
前台服务（推荐）：
    ├─ 通知始终可见
    ├─ 扫描无限制
    └─ 良好用户体验（用户看到扫描）
```

### 问题 #4：瞌睡模式 (Android 6.0+)

**症状**：应用在闲置时唤醒频率低

```
瞌睡模式：
├─ 不活跃 10 分钟后激活
├─ 禁用 WiFi、蜂窝、BLE 扫描
├─ 等待维护窗口
└─ 节电

对配网的影响：
- 瞌睡期间扫描停止
- 如果瞌睡激活无法发现设备
- 连接状态回调延迟

解决方案：
1. 请求 WAKE_LOCK 权限
   ├─ PowerManager.WakeLock
   └─ 扫描期间保持设备清醒

2. 使用 JobScheduler/WorkManager
   ├─ 系统在维护窗口中调度
   ├─ 自动批处理
   └─ 尊重瞌睡

3. 应用白名单
   ├─ 设置 → 忽略电池优化
   ├─ 用户必须手动操作
   └─ 不可靠

对于配网：
- 使用前台服务 + 通知
- BLE 在服务为前台时工作
- 良好用户体验 + 可靠配网
```

### 问题 #5：制造商特定的 BLE 行为

**症状**：制造商间行为不同

```
三星：
├─ 自定义 BluetoothGatt 扩展
├─ 有时忽略标准操作
└─ 可能需要供应商特定命令

华为：
├─ 激进的省电
├─ 扫描更积极停止
└─ 可能无故断开

索尼：
├─ 某些 Xperia 型号有 BLE 稳定性问题
└─ 需要重连逻辑

Google Pixel：
├─ 通常最标准兼容
└─ 可靠的 BLE 栈

Apple（iOS）：
├─ 某些操作需要配对
├─ 权限更严格
├─ 但稳定可靠

处理：
1. 在多个制造商上测试
2. 实现强大的错误处理
3. 使用 Nordic 库（处理很多情况）
4. 添加调试日志用于诊断
5. 考虑已知问题的设备白名单
```

### 问题 #6：服务发现时序

**症状**：连接后立即未找到服务

```
根本原因：服务发现异步
            需要 0.5-5 秒取决于：
            ├─ 设备绑定状态
            ├─ 服务数
            └─ 系统负载

问题：尝试在发现前使用服务

解决方案：Nordic 库带延迟处理
          ├─ 已绑定：发现后等待 1600ms
          ├─ 未绑定：300ms 等待
          └─ 防止时序问题

手动处理（不推荐）：
Handler().postDelayed({
    // 在这里使用服务
}, if (bonded) 1600 else 300)
```

### 问题 #7：MTU 协商失败

**症状**：MTU 保持 23 字节，尽管请求 512

```
原因：
1. 设备不支持更大 MTU
2. 系统资源耗尽
3. 固件限制

不是 bug，正常行为：
- 某些设备最多 183 字节
- 某些 250 字节
- 优雅处理任何协商的 MTU

Android 限制（9.0 之前）：
- 无法请求 MTU > 256
- Android 9+ 允许到 517

解决方案：检查协商 MTU 并适应

代码：
requestMtu(512)
    .done { mtu ->
        // mtu 可能是 23、180、250、512 等
        Log.d(TAG, "协商 MTU：$mtu")
        // 相应调整数据包大小
    }
    .enqueue()
```

## 相关协议与标准

### Matter（前身为项目 CHIP）

```
什么：Apple/Google/Amazon 智能家居开放标准

配网：
1. BLE 初始连接
2. QR 代码扫描设备设置代码
3. 设备加入 Matter fabric
4. 空中更新

WiFi 凭证：
- 配网器提供 WiFi SSID + 密码
- 设备安全存储
- 多管理员：多个家庭、访客

生态系统：
- Apple Home
- Google Home
- Amazon Alexa
- Samsung SmartThings

我们的项目 vs Matter：
- 我们：自定义 BLE 协议，特定于 WiFi
- Matter：标准协议，更广泛的 IoT 生态
- 我们：更简单，适合一次性配网
- Matter：复杂，用于广泛生态支持
- 建议：新产品使用 Matter
```

### NFC（近场通信）

```
范围：5-10 cm（接触式）
速度：106-424 kbps
用途：配对设置代码、WiFi 凭证

NFC 配网流程：
1. 用户轻触设备上的 NFC 标签
2. 手机读取标签
3. 获得设置 URL / WiFi 凭证
4. 启动配网应用
5. 应用配网设备

优势：
✓ 极短范围（无法远程拦截）
✓ 离线工作
✓ 物理确认

劣势：
✗ 需要 NFC 硬件
✗ 标签配网（一次性）
✗ 无法远程更新

组合：BLE + NFC
1. NFC 通信设备 ID
2. BLE 建立加密通道
3. BLE 发送 WiFi 凭证
4. 提供最佳安全性 + 范围
```

### WiFi Direct

```
用途：无路由器设备到设备 WiFi

配网用例：
1. 设备 A：WiFi Direct 启用
2. 设备 B：扫描 WiFi Direct 设备
3. 连接到设备 A（通过 WiFi Direct）
4. 通过 WiFi Direct 传输 WiFi 凭证
5. 设备 A 连接到实际 WiFi
6. 断开 WiFi Direct

优势：
✓ 比 BLE 更高带宽
✓ 无需 AP 的直接连接
✓ 使用现有 WiFi 栈

劣势：
✗ 并非所有设备支持
✗ 电池消耗（比 BLE 更高功耗）
✗ 用户体验（另一 WiFi 网络）

何时使用：
- 需要高带宽
- 大容量数据传输
- 仅附近设备
```

### Zigbee 和 Z-Wave

```
Zigbee：
├─ 2.4 GHz、网格网络
├─ 10-100 米范围
├─ 比 WiFi 低功耗
└─ 智能家居中常见

Z-Wave：
├─ 900 MHz（因地区而异）
├─ 网格网络
└─ 专有（由 Sigma Designs 所有）

配网：
- 也使用安全代码
- 包括带外认证
- 许多智能家居集线器使用这些

我们的项目 vs Zigbee：
- 我们：基于 BLE，使用手机作为配网器
- Zigbee：基于网格，使用集线器
- 我们：适合初次设置
- Zigbee：适合永久网络
```

## 工业配网解决方案

### Google Home / Alexa

```
Google Home（Nest）：
1. 设备上的二维码
2. 用户用手机扫描
3. 启动 Google Home 应用
4. 应用通过 BLE + WiFi 配网
5. 设备加入 Google Home 图表
6. 可远程访问

Alexa（Amazon）：
1. Alexa 应用扫描二维码
2. 提供 WiFi 网络选择
3. 通过 BLE 发送凭证
4. 设备加入 WiFi
5. 注册 Amazon 账户

安全：
✓ 使用配对加密 BLE
✓ 设备证书验证
✓ 多因素认证
✗ 专有（非开源）
```

### Apple HomeKit

```
HomeKit 安全路由（可选）：
1. USB 配件（Eve、nanoleaf 等）
2. 提供仅 HomeKit 的 WiFi 网络
3. 与主网络隔离
4. 额外安全层

配网：
1. HomeKit 应用扫描 8 位设置代码
2. 应用通过 BLE 配网设备
3. 设备加入 HomeKit 网络
4. Siri 控制立即可用

安全：
✓ ECDH 密钥交换
✓ 设备证书
✓ iCloud 凭证备份
✓ 隐私：无 Apple 遥测

实现：使用类似我们项目的协议
- BLE 传输
- 加密凭证
- 分离配网模式
```

### Samsung SmartThings

```
SmartThings 集线器：
1. 中央集线器（可选，远程访问）
2. 多协议：WiFi、Zigbee、Z-Wave、BLE
3. 云账户

配网：
1. SmartThings 应用发现设备
2. 提供 WiFi 选项
3. 用户选择网络 + 输入密码
4. 设备通过 BLE 接收凭证
5. 设备加入 WiFi 和集线器

安全：
✓ 加密通信
✓ 集线器验证
✗ 云相关（需要网络）
```

### ESP32 BluFi（Espressif）

```
什么：ESP32 的开源配网协议

协议：
1. 手机通过 BLE 发现 ESP32
2. 手机提供 WiFi SSID + 密码
3. BluFi 加密凭证 (AES-256)
4. ESP32 接收并应用
5. 固件可扩展自定义数据

实现：
公开记录在 GitHub
适合：
- 开源项目
- 自定义 IoT 设备
- 无专有限制

与我们项目的对比：
- BluFi：更成熟、测试、证明
- 我们：自定义、更简单、适合学习
- 建议：生产 ESP32 使用 BluFi
```
